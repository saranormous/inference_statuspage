How this works

Comparing reliability across AI infrastructure providers is harder than it looks. Each provider publishes a status page, but they use different platforms, report at different granularities, and define "incident" differently. A raw comparison of those numbers would be misleading.

This page uses two data sources, clearly separated:

1

Independent monitoring

Same model (Kimi K2.5), same prompt, same request — sent to each provider every 5 minutes. We measure availability, time to first token (TTFT), and total latency. Percentile stats appear after 20+ probes.

Limits: Probes run from a single location (—). Latency numbers reflect proximity to that location, not global performance. One request every 5 minutes may hit cold caches or idle GPUs — this measures availability and baseline latency, not sustained throughput. Kimi K2.5 is a thinking model, so response times include variable reasoning overhead that we don't control. Each provider may use different quantization, hardware, or batching strategies to serve the same model.
2

Self-reported incidents

We also pull each provider's public status feed and show them separately, per provider, as a record of what each has publicly disclosed. Each provider uses a different status platform, reports at different granularity, and defines "incident" differently — raw counts are not an apples-to-apples comparison.

Limits: We can only show what providers choose to report. Severity labels use different methodologies per provider.

Running the probe yourself

The monitoring probe is a small Python script that calls each provider's API and logs the result. To run it:

# Install dependencies
pip install -r monitor/requirements.txt

# Set API keys
export FIREWORKS_API_KEY=...
export TOGETHER_API_KEY=...
export BASETEN_API_KEY=...

# Run once
python monitor/probe.py

# Results append to parsed/probe_results.jsonl

A GitHub Actions workflow runs this every 5 minutes and commits results. You can also run it locally or on your own infrastructure.
